⚕ Debate: Continuwuity vs ESS Community

PROPOSER Argues FOR Continuwuity (lightweight Rust server)
OPPONENT Argues FOR ESS Community Edition (Synapse-based, Kubernetes)

Topic: Which Matrix homeserver should the Shape Rotator cohort deploy? ~20-50 crypto/TEE researchers on modest hardware (2-4 cores, 8-16GB RAM). Must federate with matrix.org. Limited admin time — prefer "set it and forget it."

Round 1

Proposer Continuwuity

Position: Continuwuity is the better choice for modest hardware, limited admin time, and a 20-50 user research cohort.

  • Single Rust binary + RocksDB = ~500MB DB, negligible RAM/CPU on 2-4 core VPS
  • One TOML config file, no Kubernetes, no PostgreSQL — true "set it and forget it"
  • Privacy-focused defaults: federated presence off by default, CIDR denylist for outbound requests
  • E2EE with cross-signing, key backup, dehydrated devices (MSC3814)
  • Appservice protocol for bridging to Discord/Telegram via mautrix
  • MatrixRTC support for VoIP via external LiveKit

Source: Comparison report Section 3 — "community reports cite ~500 MB of DB and barely noticeable RAM/CPU"

Opponent ESS Community

Position: ESS Community Edition provides a complete, battle-tested stack with bundled services.

  • Bundled Element Call + LiveKit SFU — video calling works out of the box, no separate deployment
  • MAS provides full OIDC/SSO with Keycloak, Dex, etc.
  • Synapse = reference homeserver, best Matrix spec compliance
  • Hookshot bundled for integrations since 26.1.1
  • Element Admin web console for user/room management

Source: Comparison report Section 1 — "Matrix 2.0 (OIDC-native, Simplified Sliding Sync, Element Call) — Full for ESS"

Round 2 — Rebuttals

Proposer Rebuttal

  • ESS Community's Kubernetes requirement creates disproportionate complexity for a single VPS — Continuwuity runs standalone
  • ESS's "2 CPU / 2 GB RAM minimum" is just for the stack itself, leaving no headroom for actual traffic on a shared VPS
  • External LiveKit is a one-time setup; the team has TEE and crypto expertise — not a barrier

Opponent Rebuttal

  • Continuwuity lacks native OIDC — only legacy auth (LDAP, reCAPTCHA, registration tokens)
  • No bundled bridges — each mautrix bridge is a separate deploy/maintain cycle
  • ESS Community's Helm chart automates updates and scaling; Continuwuity has no horizontal scale-out story at all
  • For 20-50 users, ESS's resource requirements are manageable even on modest hardware

Verdict

WINNER: PROPOSER Continuwuity

8 — 6
"The 'set it and forget it' requirement combined with modest hardware makes Kubernetes fundamentally wrong-headed. Continuwuity's single-binary architecture aligns perfectly with the constraints; ESS Community forces distributed system complexity that the use case cannot justify."

Reasoning

Continuwuity aligns perfectly with the constraints: single Rust binary designed for modest hardware (~500MB DB, negligible RAM/CPU), single TOML config file for true "set it and forget it" administration, no Kubernetes or PostgreSQL complexity, reliable federation with proper controls, full E2EE support including dehydrated devices.

Recommendation: Deploy Continuwuity using Docker or systemd for process management. Configure federation and appservice bridges (mautrix-discord, mautrix-telegram) as separate containers. If group video calls become essential, deploy LiveKit + lk-jwt-service alongside. This minimizes admin burden while maximizing resilience.
